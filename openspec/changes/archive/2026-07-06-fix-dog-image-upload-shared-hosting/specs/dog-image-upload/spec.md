## MODIFIED Requirements

### Requirement: Dog profile image upload accepts files up to 5 MB
The system SHALL accept dog profile image uploads up to the Laravel validation limit
of 5 MB (5120 KB) without rejecting them at the web-server layer, **in every deployed
environment** (local Docker/nginx, and Apache-based shared hosting for demo/production).

The nginx reverse proxy (local Docker environment) MUST be configured with
`client_max_body_size 10M` (2× the Laravel validation limit) so that multipart
overhead does not cause nginx to reject valid uploads before the request reaches
Laravel.

The Apache configuration shipped for shared hosting
(`deployment-templates/htaccess/backend-public.htaccess`, copied to
`backend/public/.htaccess` by the deployment build scripts) MUST set
`LimitRequestBody 10485760` (10 MB, same 2× buffer convention as nginx) so that
Apache itself does not reject the request body before it reaches PHP/Laravel.
Additionally, the shipped PHP configuration overrides
(`php_value upload_max_filesize`/`post_max_size` in `.htaccess` for `mod_php` hosts;
`.user.ini` for PHP-FPM hosts; and `php.ini` as an additional fallback for
CGI/FastCGI hosts where `.user.ini` is not evaluated, since the exact PHP
execution mode of the target shared-hosting provider is not known) MUST allow
at least 10 MB uploads / 12 MB POST bodies, so that the default PHP ini limits
of the hosting provider do not silently truncate or reject valid uploads before
Laravel's own validation runs. Both the `.user.ini` and `php.ini` files MUST be
delivered to the built deployment package by **every** packaging path that
produces a shared-hosting artifact — `build-deployment.sh`,
`build-deployment-docker.sh`, and the CI-driven `.github/workflows/deploy.yml`
packaging step — so that none of them silently omits the PHP-FPM/CGI-FastCGI
fallback while still shipping the `.htaccess` (T01) mechanism.

Because `.user.ini` and `php.ini` are ordinary text files (unlike `.htaccess`,
which Apache denies by its own built-in `.ht*` default), the shipped
`backend-public.htaccess` MUST also deny direct public HTTP access to both
files, so that their contents are not served as plain text to arbitrary
clients.

When a file exceeds the Laravel validation limit (> 5 MB), Laravel SHALL return
HTTP 422 with a structured validation error. The CORS headers SHALL be present in
the response in all cases (neither nginx nor Apache/PHP intercept requests below
their respective body-size limits).

#### Scenario: Upload of a valid image (≤ 5 MB) succeeds
- **WHEN** a user submits a dog profile image ≤ 5 MB via `DogFormModal`
- **THEN** the system returns HTTP 200 and the dog's `profile_image` field is updated

#### Scenario: Upload rejected by Laravel validation (> 5 MB)
- **WHEN** a user submits a dog profile image > 5 MB
- **THEN** Laravel returns HTTP 422 with a `errors.image` validation message
- **THEN** CORS headers (`Access-Control-Allow-Origin`) are present in the response

#### Scenario: Upload blocked by nginx (> 10 MB, local Docker environment)
- **WHEN** a user submits a file > 10 MB against the local Docker/nginx stack
- **THEN** nginx returns HTTP 413
- **THEN** the browser does not report a spurious CORS error for a < 10 MB file

#### Scenario: Upload blocked by Apache (> 10 MB, shared hosting)
- **WHEN** a user submits a file > 10 MB against a shared-hosting deployment
  (Apache with `backend/public/.htaccess` as generated by the build scripts)
- **THEN** Apache returns HTTP 413 (via `LimitRequestBody`)
- **THEN** the browser does not report a spurious CORS error for a < 10 MB file

#### Scenario: Upload of a 2-8 MB image succeeds on shared hosting
- **WHEN** a user submits a dog profile image between 2 MB and 8 MB against a
  shared-hosting deployment whose PHP defaults (`upload_max_filesize`/
  `post_max_size`) would otherwise be below that size
- **THEN** the request reaches Laravel (neither Apache's `LimitRequestBody` nor
  the shipped PHP ini overrides reject it before Laravel's own `max:5120`
  validation runs)
- **THEN** the system returns HTTP 200 and the dog's `profile_image` field is updated

#### Scenario: False CORS error no longer occurs for images between 1 MB and 10 MB
- **WHEN** a user uploads a file between 1 MB and 10 MB, in any deployed
  environment (local Docker/nginx or shared hosting/Apache)
- **THEN** the request reaches Laravel (neither nginx nor Apache intercept with
  HTTP 413)
- **THEN** no `Access-Control-Allow-Origin` error is raised in the browser console

#### Scenario: PHP config override files are not exposed via public HTTP access
- **WHEN** a client requests `GET /.user.ini` or `GET /php.ini` directly against
  a shared-hosting deployment (`backend/public/.user.ini` / `backend/public/php.ini`,
  as generated by the build scripts)
- **THEN** Apache returns HTTP 403 (via the `<FilesMatch>` deny block in
  `backend-public.htaccess`), not the plain-text file contents
- **THEN** the existing rewrite fallback to `index.php` for non-existent paths,
  and the existing security headers, remain unaffected

## ADDED Requirements

### Requirement: Failed profile image uploads are visibly reported and do not silently close the form
The system SHALL clearly and persistently report a failed profile image upload to the user, and SHALL NOT close `DogFormModal` as if the entire save operation succeeded.

This applies whenever the profile image upload sub-step (`POST /api/v1/dogs/{id}/upload-image`,
triggered from `DogFormModal` after the dog's core data has been saved) fails for
any reason (validation error, server error, network error, or a web-server-level
rejection such as HTTP 413).

The system SHALL NOT create a duplicate dog record if the user retries saving
after a core-data save succeeded but the subsequent image upload failed.

#### Scenario: Image upload fails after a successful core-data save (edit flow)
- **WHEN** a user edits an existing dog, the core-data save succeeds, and the
  subsequent image upload fails
- **THEN** the form (`DogFormModal`) remains open
- **THEN** a persistent, visible error message is shown (not only a transient toast)
- **THEN** the user can retry uploading the image without re-entering the other form fields

#### Scenario: Image upload fails after a successful core-data save (create flow)
- **WHEN** a user creates a new dog, the core-data save succeeds (a new dog record
  is created), and the subsequent image upload fails
- **THEN** the form remains open with a persistent, visible error message
- **THEN** retrying the save does NOT create a second dog record

#### Scenario: Image upload succeeds
- **WHEN** the image upload sub-step succeeds
- **THEN** the form closes and a success confirmation is shown, as before this change
